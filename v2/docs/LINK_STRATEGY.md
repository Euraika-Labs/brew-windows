# Windows Link Strategy

This is the most upstream-relevant document in v2. It describes the
single largest patch we maintain against `Homebrew/brew` and the
shape of the eventual upstream PR.

The decision itself is recorded in
[ADR 0005](adr/0005-windows-link-strategy.md). This document is the
implementation spec.

## The Problem

Homebrew's install flow ends with a "link" step. For every executable
that lives in `Cellar/<name>/<version>/bin/<exe>`, Homebrew creates a
symlink at `<prefix>/bin/<exe>` pointing back into the Cellar.

On macOS and Linux, symlinks are free and unprivileged. On Windows:

- `CreateSymbolicLinkW` requires the `SeCreateSymbolicLink` user right.
  Unprivileged users do not have this by default.
- Developer Mode (Windows 10+) grants the right without elevation, but
  Developer Mode is a setting users have to enable.
- Some Windows policies (managed devices, antivirus, EDR) refuse
  symlinks regardless.

v2's design constraint is **no elevation, no Developer Mode required,
no group policy changes**. So symlinks are not an option for the
default install.

## The v1 Proof

v1 demonstrated a working alternative: generate a `.cmd` + `.ps1` pair
per executable that delegates to the real `.exe` in the Cellar.

```cmd
@echo off
powershell.exe -NoProfile -ExecutionPolicy Bypass -File "%~dp0<name>.ps1" %*
exit /b %ERRORLEVEL%
```

```powershell
# Generated by Brew Windows. Do not edit.
$ErrorActionPreference = "Stop"
$target = 'C:\Users\...\Cellar\<name>\<ver>\bin\<exe>'
& $target @args
exit $LASTEXITCODE
```

`v1/tests/shim-fuzz.ps1` validated that:

- Plain text arguments round-trip.
- Arguments with spaces round-trip (`hello world`).
- Symbol-laden arguments round-trip (`paren(value)`, `bracket[1]`,
  `amp&value`, `caret^value`).
- Single quotes round-trip.
- Paths with spaces round-trip.
- The shim's exit code matches the target's exit code.

Those tests are the spec. Any v2 link strategy implementation must
pass them.

## The v2 Patch

The patch teaches Homebrew's keg linking to use this shim strategy on
Windows.

### Patch shape

Target file: `Library/Homebrew/keg.rb` (or
`Library/Homebrew/extend/os/windows/keg.rb` if upstream prefers the
`extend/os/` pattern they already use for macOS-specific behavior).

```ruby
# Library/Homebrew/extend/os/windows/keg.rb (new file)

module Homebrew
  module ExtendOS
    module Windows
      module Keg
        # Override link strategy on Windows: emit .cmd + .ps1 shims
        # instead of symlinks for executables under bin/.

        def link_dir(relative_dir, **options)
          if relative_dir.to_s == "bin"
            link_bin_dir_with_shims(**options)
          else
            super
          end
        end

        private

        def link_bin_dir_with_shims(**options)
          target_dir = HOMEBREW_PREFIX / "bin"
          target_dir.mkpath

          (path / "bin").children.each do |source|
            next unless source.file?
            next unless windows_executable?(source)

            name = source.basename(".exe").to_s
            generate_shim(target_dir, name, source)
          end
        end

        def windows_executable?(path)
          %w[.exe .cmd .bat].include?(path.extname.downcase)
        end

        def generate_shim(target_dir, name, source)
          cmd_path = target_dir / "#{name}.cmd"
          ps1_path = target_dir / "#{name}.ps1"

          cmd_path.atomic_write(<<~CMD)
            @echo off
            powershell.exe -NoProfile -ExecutionPolicy Bypass -File "%~dp0#{name}.ps1" %*
            exit /b %ERRORLEVEL%
          CMD

          escaped_target = source.to_s.gsub("'", "''")
          ps1_path.atomic_write(<<~PS1)
            # Generated by Brew Windows. Do not edit.
            $ErrorActionPreference = "Stop"
            $target = '#{escaped_target}'
            & $target @args
            exit $LASTEXITCODE
          PS1
        end
      end
    end
  end
end

Keg.prepend(Homebrew::ExtendOS::Windows::Keg) if OS.windows?
```

The pattern (`module ExtendOS::Windows::Keg`, `Keg.prepend`,
guarded by `OS.windows?`) is the same shape upstream already uses
for macOS and Linux extensions in `Library/Homebrew/extend/os/`. The
patch contributes a third sibling, not a structural change.

### What the patch does not touch

- `link` for non-`bin/` directories. `share/`, `etc/`, `lib/`, `include/`
  all still go through whatever upstream does. (For most CLI tools we
  install in Phase 3 there is nothing to link in those.)
- `unlink`. The default symlink-aware `unlink` works for `.cmd` and
  `.ps1` files because they are plain files - `unlink` just deletes
  them.
- Conflict detection. Existing upstream code that refuses to
  overwrite a non-symlink target also refuses to overwrite an
  existing shim. Good.

### The os.windows? prerequisite

The patch above guards with `OS.windows?`. That method does not yet
exist in `Library/Homebrew/os.rb`. A separate, smaller patch adds it:

```ruby
# Library/Homebrew/os.rb (existing file, ~10 line patch)

module OS
  def self.mac?
    # ... unchanged ...
  end

  def self.linux?
    # ... unchanged ...
  end

  def self.windows?
    RbConfig::CONFIG["host_os"].match?(/mswin|mingw|cygwin/)
  end
end
```

RubyInstaller's `host_os` is `mingw32` or `mingw64`, so the regex
matches. Macropolitan minimal change.

This `OS.windows?` patch is the first PR candidate in the eventual
upstream sequence, identified by MikeMcQuaid as plausible if all the
other prerequisites are met.

## Total Patch Size

Phase 3 working estimate:

- `windows-os-detection.patch` - ~30 lines (`OS.windows?` + a couple of
  `extend/os/windows/...` extension stubs).
- `windows-link-strategy.patch` - ~150 lines (the keg link override
  above plus tests).

Combined: ~180 lines. Well under MikeMcQuaid's <500 LOC PR target.

## Behavior Specification (Spec, Not Aspirational)

Every shim generated by the patch MUST:

1. Pass v1's `shim-fuzz.ps1` argument fuzz suite.
2. Propagate the target's exit code exactly.
3. Survive being placed in a prefix whose path contains spaces.
4. Survive being placed in a prefix whose path contains `[`, `]`,
   parentheses, or commas.
5. Tolerate the target being a `.exe`, a `.cmd`, or a `.bat`.
6. Not modify the target's arguments (no quote-mangling, no
   word-splitting).
7. Refuse to overwrite a file that is not a brew-generated shim
   (`Generated by Brew Windows` marker).

If any of these fails, the patch is wrong.

## Conflict Detection

Upstream Homebrew already refuses to clobber a non-symlink at the
target path. The patch retains that behavior:

- If `<prefix>/bin/<name>.cmd` exists and does not contain the
  `Generated by Brew Windows` marker, the link step throws.
- Same for `<prefix>/bin/<name>.ps1`.
- Same for `<prefix>/bin/<name>.exe` (the user has installed something
  else that conflicts).

Pre-existing conflicts are surfaced clearly:

```
Error: cannot link bin/codex - <prefix>\bin\codex.cmd already exists
and is not managed by Brew Windows.
Run `brew link --overwrite codex` to replace it.
```

## Doctor Integration

`brew doctor` (Phase 2) gains a Windows-specific check:

- For every receipt in `<prefix>/var/homebrew/receipts/`, verify that
  the matching shim pair in `<prefix>/bin/` exists and points at a
  Cellar path that still exists.
- Warn on dangling shims (target Cellar path missing).
- Warn on shadowed shims (the same name resolves to a different
  binary earlier in PATH).

These match v1's `brew doctor` warnings.

## Long Term Upstream Path

The PR sequence after Phase 4:

1. `OS.windows?` patch as a single small PR. Useful or inert on
   macOS/Linux. No behavior change.
2. The keg link strategy abstraction: extract symlink logic into a
   `PosixSymlinkStrategy` class, default unchanged, expose a hook for
   alternative strategies. No behavior change for macOS/Linux.
3. The Windows shim strategy as a follow-up PR using the hook from (2).
   Guarded by `OS.windows?`. Initially behind an unsupported/experimental
   flag.

(1) and (2) are useful even if Windows support never lands - they
clarify upstream code. (3) is the actual Windows enablement.

## Open Questions

- **Renaming vs in-place rewrite during upgrade**. When `brew upgrade
  <name>` builds a new keg, the old keg's shims need to point at the
  new one. Upstream's current flow: unlink old, link new. Same flow
  works for shims (delete + recreate). Verify in Phase 3.
- **What about scripts that need to be invoked as `python` not
  `python.exe`?** Some Homebrew formulae are Python-script
  shebang-style. Windows does not honor shebangs. v2 likely needs
  a small `<name>.ps1` shim that knows to invoke a specific
  interpreter for shebang scripts. Phase 3 surface area.
- **What about formulae that install dylibs / DLLs to `lib/`?**
  Most Phase 3 candidates are statically-linked Rust/Go binaries with
  no `lib/` content. When we hit a formula with `lib/` content,
  decide whether to also generate stubs there or rely on upstream's
  default. Phase 3 surface area.
