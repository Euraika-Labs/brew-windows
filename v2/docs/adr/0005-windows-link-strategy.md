# v2 ADR 0005: Windows Link Strategy Uses .cmd + .ps1 Shim Pairs

Date: 2026-05-24

## Status

Accepted.

## Context

Homebrew's install pipeline ends with a "link" step that exposes
installed binaries on the user's PATH. On macOS and Linux this is done
with symlinks from `<prefix>/bin/<name>` to
`<prefix>/Cellar/<name>/<ver>/bin/<name>`.

Symlinks on Windows are not the default tool for this job. They
require either:

- The `SeCreateSymbolicLink` user right (not granted by default), or
- Developer Mode enabled (Windows 10+; a deliberate user-action
  setting), or
- An elevated process.

v2's design constraint is **no elevation, no Developer Mode required,
no group policy changes**. Symlinks therefore cannot be the link
strategy.

v1 demonstrated an alternative that works without any of these
requirements: generate a `.cmd` + `.ps1` shim pair per executable that
delegates to the real binary in the Cellar.

The v1 shim format is:

```cmd
@echo off
powershell.exe -NoProfile -ExecutionPolicy Bypass -File "%~dp0<name>.ps1" %*
exit /b %ERRORLEVEL%
```

```powershell
# Generated by Brew Windows. Do not edit.
$ErrorActionPreference = "Stop"
$target = '<cellar-path-to-executable>'
& $target @args
exit $LASTEXITCODE
```

v1's `tests/shim-fuzz.ps1` proved the shim format preserves arguments
across both shim variants under adversarial inputs and propagates
exit codes correctly.

## Decision

v2 adopts the v1 shim format unchanged as the Windows link strategy.

- For every executable in `<prefix>/Cellar/<name>/<ver>/bin/`, the
  Windows link step writes a `.cmd` + `.ps1` pair to
  `<prefix>/bin/`.
- The `.ps1` is the actual launcher. The `.cmd` is the
  execution-policy-tolerant front door.
- Both shims contain a fixed marker (`Generated by Brew Windows. Do
  not edit.`) so doctor checks and conflict detection can distinguish
  brew-managed files from user-placed ones.
- Conflict detection refuses to overwrite a non-marker file.
- v1's `shim-fuzz.ps1` test suite is the spec for v2's link strategy
  patch. Any change to the shim format must continue to pass the
  full suite.

This logic lives in a Ruby patch against upstream Homebrew
(`patches/windows-link-strategy.patch`), guarded by `OS.windows?`.
See [LINK_STRATEGY.md](../LINK_STRATEGY.md) for the implementation.

## Consequences

### Functional

- A user runs `brew install codex`. After install, `codex.exe` is in
  the Cellar; `codex.cmd` and `codex.ps1` are in `<prefix>/bin/`.
  The user types `codex` in PowerShell and gets the right binary.
- Arguments with spaces, parens, brackets, ampersands, single
  quotes, paths-with-spaces all round-trip correctly.
- Exit codes propagate exactly.
- Pipes (`echo y | codex something`) work because PowerShell's `&`
  preserves stdin.

### Operational

- No admin rights ever required for any install.
- No Developer Mode required.
- No group policy changes required.
- No registry writes required.
- Works under restricted PowerShell execution policy because the
  `.cmd` front door passes `-ExecutionPolicy Bypass` to its embedded
  `.ps1`.

### Performance

- A shim invocation costs one `cmd.exe` -> `powershell.exe` -> target
  hop. Cold start of PowerShell adds ~200-500 ms on first invocation.
  Subsequent invocations from the same shell are warm.
- For CLI tools used many times per shell session (e.g. `rg` in an
  interactive loop), the cold-start cost happens once.
- Alternative: a hand-written `.exe` shim (e.g. via a small Go or
  Rust binary). Faster cold start but adds a compilation step to
  install, and is meaningfully more complex. Deferred until Phase 4+
  if cold-start cost is a real complaint.

### Visibility

- Installed binaries appear as `<name>.cmd` and `<name>.ps1` in
  PowerShell tab-completion, instead of `<name>.exe`. Users see
  `gh.cmd`, not `gh.exe`. This is normal on Windows and rarely
  surprising; many third-party tools (scoop, chocolatey, npm
  global installs) do similar.

## Conflict Detection

The link patch refuses to overwrite a file at `<prefix>/bin/<name>.cmd`
or `<prefix>/bin/<name>.ps1` if:

- The file exists and does not contain the
  `Generated by Brew Windows. Do not edit.` marker.

This protects:

- User-placed scripts in `<prefix>/bin/`.
- Other tools that wrote files into the prefix manually.

Conflict resolution: the user passes `brew link --overwrite <name>`,
matching upstream Homebrew's existing flag.

## Unlink Semantics

Upstream Homebrew's default unlink path deletes the symlink. Our
shims are plain files; the same delete works. No special unlink
override required.

The patch verifies marker presence before unlinking. Refusing to
unlink a file without the marker prevents the patch from accidentally
deleting a user file even when invoked through `brew unlink`.

## Doctor Integration

`brew doctor` (Phase 2) checks:

- Every receipt has a matching `.cmd` + `.ps1` pair in
  `<prefix>/bin/`.
- Every `.cmd` and `.ps1` in `<prefix>/bin/` either:
  - Contains the marker AND points at an extant Cellar path, OR
  - Has no marker (= user-placed file; doctor ignores).
- Every shim that has the marker but points at a missing Cellar
  path is reported as broken.
- Every brew-managed shim that is shadowed by an earlier-PATH file
  is warned about.

These checks come straight from v1's doctor.

## Rejected Alternative: Symlinks With Developer Mode

Rejected because:

- Requires the user to enable Developer Mode, which is a
  Settings-app-toggle action.
- Even with Developer Mode, some EDR / antivirus products block
  symlink creation.
- Some users on managed devices cannot enable Developer Mode.
- Setting this requirement contradicts the "no policy changes"
  design constraint.

## Rejected Alternative: Junctions

Rejected because:

- Directory junctions work without elevation but file junctions do
  not exist. Homebrew's link step links *files* in `bin/`, not
  directories.
- Junctioning the entire `bin/` directory of a Cellar entry into
  `<prefix>/bin/` would conflict with the next install.

## Rejected Alternative: Hard Links

Rejected because:

- Hard links require source and target on the same filesystem.
  Per-user prefix and Cellar are on the same volume, so this would
  *work*, but:
- Hard links require write access to both source and target. This
  is the case for us, but the resulting behavior surprises users:
  modifying the file at `<prefix>/bin/codex` also modifies the file
  in the Cellar.
- Some antivirus tools flag hard links into user-writable directories
  as suspicious.
- Hard links don't survive across formula upgrades cleanly without
  unlink + relink, which is what shims already do.

## Rejected Alternative: Native `.exe` Shim

Rejected for now; revisited Phase 4+ if needed.

A small `.exe` shim (compiled Go, Rust, or even C) would have lower
cold-start cost than the PowerShell variant. But:

- Adds a build artifact to the v2 release pipeline.
- Adds a compiled binary to scrutinize (Authenticode signing
  question, antivirus false positives).
- The cold-start cost has not been a real user complaint - PowerShell
  is the user's shell, so PowerShell is already warm.

If Phase 4 produces consistent complaints about cold start, this
ADR will be revisited.

## Upstream PR Shape

The Phase 5 upstream PR sequence is built around this strategy:

1. PR 1: `OS.windows?` predicate (prerequisite).
2. PR 2: `Library/Homebrew/extend/os/windows/` extension point.
3. PR 3: Keg link extraction into a `PosixSymlinkStrategy` class
   (no behavior change for macOS/Linux).
4. PR 4: Windows shim link strategy, using the hook from PR 3,
   guarded by `OS.windows?` and an experimental flag.

PR 4 carries the format defined here. The fuzz test suite (v1's
`shim-fuzz.ps1` ported to Ruby + bats or RSpec) is the contract.

## Open Questions

These do not affect the decision but will surface during
implementation:

- Shebang scripts. Some Homebrew formulae install Python or Ruby
  scripts with shebang lines and no `.exe` extension. Windows does
  not honor shebangs. v2 needs an additional shim form that knows
  to invoke a specific interpreter. Phase 3 surface area.
- Manpages, completions, info pages. These do not need shims (they
  are read, not executed). They stay as files in
  `<prefix>/share/man/`, `<prefix>/share/zsh/`, etc. Discoverability
  on Windows is a separate question deferred to Phase 4.
- DLL placement. Most Phase 3 candidate formulae are statically
  linked. When we hit a formula that ships DLLs, decide whether to
  also shim `lib/`. Phase 3 surface area.
